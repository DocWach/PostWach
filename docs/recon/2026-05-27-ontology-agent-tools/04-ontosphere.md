Recon date: 2026-05-27. Source: shallow local clone at `C:/Users/pfwac/recon-scratch/ontosphere` (branch `main`, HEAD `f1d5240`, last commit 2026-05-22). All claims below are verified against the clone unless tagged `[unverified]`.

## What it is

ontosphere is a browser-based RDF (Resource Description Framework) / ontology knowledge-graph editor that runs entirely client-side: no backend, an in-memory RDF store backed by Web Workers, OWL 2 DL (Web Ontology Language, Description Logic profile) reasoning, multi-algorithm layout, and a Model Context Protocol (MCP) tool surface for AI-agent control. It loads RDF from local files, remote URLs, or SPARQL (SPARQL Protocol and RDF Query Language) / Fuseki endpoints; lets users author nodes and edges directly on a canvas; and exports Turtle, JSON-LD, or RDF/XML. Author is Thomas Hanke (ORCID 0000-0003-1649-6832), affiliated with Fraunhofer Institute for Mechanics of Materials IWM. The "AI-native" framing is real and central, not marketing veneer: the MCP surface is a first-class, designed subsystem (see below), not an afterthought.

## Feature set

- **RDF loading:** local files, remote URLs, and SPARQL/Fuseki endpoints; URL params `?url=`, `?ontology=foaf`, `?loadImports=false`. A registry of ~55 well-known ontologies (FOAF pre-loaded; OWL/RDFS/RDF/XSD always available) with prefix-based loading and use-case search.
- **Two-view canvas authoring:** separate ABox (individuals/instances) and TBox (classes/properties) views; nodes/edges authored directly; OWL restrictions authorable via blank nodes. Node/edge mutations write to an N3 store that is the source of truth; the Reactodia canvas mirrors it.
- **Reasoning:** OWL 2 DL via Konclude (default) or N3 rule-based inference; inferred triples rendered as amber dashed edges with a reasoning report; inferred quads kept in a separate named graph so they can be cleared without touching asserted data.
- **SHACL validation:** load shapes, validate the data graph, get a structured violation list.
- **Layout/clustering:** Dagre (LR/TB) and ELK (layered, force, stress, radial); Louvain / SLPA / coarsen / k-means clustering for large graphs.
- **Namespace management:** live prefix upsert/remove/list with URI renaming.
- **Export:** Turtle, JSON-LD, RDF/XML for data; SVG (token-efficient) or PNG for canvas images, with an optional focus-IRI clipped viewBox.
- **Named-graph architecture (verified in `src/mcp/manifest.ts`):** `urn:vg:data` = asserted triples, `urn:vg:inferred` = DL-derived, `urn:vg:shapes` = SHACL constraints. This three-graph separation is clean and is exactly the discipline we want for a governance ontology (asserted vs. inferred vs. constraint).

## MCP tool surface & relay bridge (DEEP)

This is the strategically important part, and it is more substantial than the brief implied.

**Tool surface (verified, 35 tools).** The full MCP manifest lives in `src/mcp/manifest.ts`; handlers are in `src/mcp/tools/{graph,nodes,links,layout,reasoning,namespaceTools,navigation,shacl}.ts`, aggregated in `src/mcp/ontosphereMcpServer.ts`. The exact tool names (do not paraphrase these): `loadRdf`, `loadOntology`, `suggestOntologiesForTask`, `queryGraph`, `exportGraph`, `exportImage`, `setViewMode`, `addNode`, `removeNode`, `expandNode`, `getNodes`, `addTriple`, `removeLink`, `getLinks`, `runLayout`, `focusNode`, `fitCanvas`, `runReasoning`, `clearInferred`, `getCapabilities`, `getGraphState`, `setNamespace`, `removeNamespace`, `listNamespaces`, `getNodeDetails`, `updateNode`, `getNeighbors`, `findPath`, `loadShacl`, `validateGraph`, `help`. Each carries a JSON Schema `inputSchema`. `queryGraph` is a full SPARQL read/update endpoint (SELECT, CONSTRUCT, INSERT/DELETE DATA, DELETE WHERE, DELETE...INSERT...WHERE) over the in-browser store. The manifest's `mcpServerDescription` is a written-for-LLMs operating manual (workflow ordering, batching rules, blank-node conventions), which is a notable design pattern in itself.

**How tools are exposed (three transports, verified in `src/mcp/ontosphereMcpServer.ts`, `register.ts`, `workspaceContext.ts`):**
1. **`window.__mcpTools` (in-page).** `registerMcpTools()` always builds a `toolMap` of name -> schema-validated handler and assigns it to `window.__mcpTools`, regardless of browser support. This is the substrate everything else rides on.
2. **`navigator.modelContext` (native browser MCP).** If the browser exposes the experimental `navigator.modelContext` API, each tool is also registered via `mc.registerTool(name, description, inputSchema, handler)`. In headless Chromium this API is absent, so Playwright injects a polyfill via `page.addInitScript` (documented in the README) and then drives `window.__mcpTools[name](params)` directly. This is the path a Claude Code / computer-use agent would take today.
3. **AI Relay Bridge (the novel part).** For hosted chat UIs (ChatGPT, Gemini, Claude.ai) where you cannot inject scripts at load time, a **bookmarklet** runs in the AI chat tab and scans the model's visible output for backtick-wrapped JSON-RPC 2.0 `tools/call` messages. The data path (from `docs/relay-bridge.md` and `src/mcp/relayBridge.ts`):

   `AI chat tab --postMessage(vg-call)--> relay.html popup --BroadcastChannel(vg-call)--> Ontosphere tab --executes tool--> BroadcastChannel(vg-result) --> relay popup --postMessage(vg-result)--> AI chat tab (result auto-injected into the chat input and submitted).`

   The cross-tab transport is a `BroadcastChannel` named `ontosphere-relay-v1`. `relayBridge.ts` implements: a message protocol (`vg-call`, `vg-result`, `vg-ping`, `vg-ready`), a per-page-load `SESSION_ID` to detect reloads, a 10-second heartbeat ping during long tools (layout/reasoning), connection liveness with a 15 s stale timeout, an internal serial dispatch queue (calls run one at a time; up to 5 batched per relay message per the manifest's guidance), and a best-effort SVG export appended to the last call of a batch so the model "sees" the resulting diagram. `workspaceContext.ts` answers the startup ping with `vg-ready` once the Reactodia workspace and reasoning callback are wired, giving a clean readiness handshake.

**Architectural read.** The relay bridge is essentially a zero-install, no-extension, no-server way to give a sandboxed chat LLM a tool-calling channel into a live browser app, using JSON-RPC over `BroadcastChannel` + `postMessage` and screen-scraping the model's own text output for tool calls. It is clever and genuinely portable across AI vendors. It is also inherently fragile (depends on the chat site's DOM, popup permissions, and the model emitting exactly-formatted backtick JSON) and is the wrong primitive for *our* hives, where coordinator agents already speak MCP over stdio/socket via claude-flow. The reusable idea for us is not the bookmarklet; it is the **`window.__mcpTools` + `navigator.modelContext` + named-graph (`urn:vg:*`) pattern**: a clean separation between (a) a transport-agnostic in-process tool map, (b) optional native-MCP registration, and (c) a typed, schema-validated handler layer over a triple store with asserted/inferred/shapes graphs. There is no consumption of external MCP servers; ontosphere is purely an MCP *provider*.

## OWL-RL reasoning

The brief asked about "OWL-RL," but ontosphere does not implement the OWL 2 RL profile. It offers two reasoner backends, selectable via `runReasoning({reasonerBackend})` (verified in `src/mcp/tools/reasoning.ts` and `src/workers/rdfManager.runtime.ts`):

- **Konclude (default).** `rdf-reasoner-konclude` v0.2.1 wraps the Konclude reasoner, a complete tableau reasoner for the description logic SROIQ(D), i.e. full **OWL 2 DL**, compiled to WebAssembly. It runs in a dedicated Web Worker (`public/rdf-reasoner-konclude/worker.js`), performs classification, and writes `rdfs:subClassOf` / `owl:equivalentClass` inferences back into `urn:vg:inferred`. Because the WASM uses `SharedArrayBuffer` (pthreads), deployment requires cross-origin isolation headers (`Cross-Origin-Opener-Policy: same-origin`, `Cross-Origin-Embedder-Policy: credentialless`); `server.js` sets these, and localhost has SharedArrayBuffer without them. So the "fully client-side" claim is accurate for compute, but production hosting needs COOP/COEP headers, a real (if minor) deployment constraint.
- **N3 rule reasoner.** The `Reasoner` class bundled in the `n3` library (n3.js), used for N3/rule-based inference. Lighter weight, no WASM/header requirement.

Net: client-side OWL 2 DL classification in the browser via WASM is the headline capability, and it is real (R016 status (b), see below). "OWL-RL" specifically is `[unverified]` and appears to be a mislabel in the brief.

## Adopt-into-hive assessment (R016-tagged)

**(i) As a visual editor/inspector for our portfolio governance ontology.** Candidate: **R016 (b) demonstrated capability**, adoptable at low cost. ontosphere can load our `portfolio-governance.ttl` (TBox), `portfolio-abox.ttl` (ABox), and `portfolio-shacl.ttl` (shapes) from files or a Fuseki endpoint, render the TBox/ABox split, run our SHACL shapes via `loadShacl`/`validateGraph`, and run DL classification to sanity-check the class hierarchy visually. It is a complementary inspector/authoring surface, not a replacement for the GI-JOE ontology-gate pipeline (rdflib + pyshacl + SPARQL CQ manifest), which stays canonical. Integration cost: low. We already run a local Fuseki; point `?url=` or the load tools at it, or drag the .ttl files in. The main caveats are (1) it is an in-memory editor with no persistence back to our repo (export-to-Turtle then commit is a manual round-trip), and (2) our SPARQL competency questions (CQs) are SPARQL-1.1 over named graphs and would run through `queryGraph`, but the canonical CQ run should remain the repo's manifest-driven verifier. Status if adopted as a portfolio asset: this would be a **(b) demonstrated capability** (a tool shown to work in isolation against our files), and must not be presented as a **(c) integrated deliverable** unless/until it is wired into the gate workflow.

**(ii) As a model for exposing ontology operations to claude-flow agents via MCP.** Candidate: **R016 (a) research artifact / pattern reference**, medium adoption value. The valuable, transferable architecture is the **typed tool layer over a triple store with `urn:vg:data` / `urn:vg:inferred` / `urn:vg:shapes` separation**, schema-validated handlers, and a single in-process tool map. We could mirror this as an MCP server our coordinator agents call over claude-flow's existing transport: tools like `queryGraph`, `validateGraph`, `runReasoning`, `addTriple`, `getNeighbors`, `findPath` map almost one-to-one onto governance-ontology operations we already do by hand. We would NOT adopt the relay bookmarklet; our agents already have a real MCP transport, so the relay-bridge layer is solving a problem (sandboxed chat tabs) we do not have. Integration cost to copy the *pattern*: medium (re-implement the handler layer against our store/endpoint, ignore the relay). Integration cost to *reuse ontosphere as-is as our MCP server*: high and not recommended, because its tools are bound to an in-memory browser store and a Reactodia canvas, not a server-side Fuseki, so it cannot serve as a headless ontology MCP server without significant rework.

**Relay-bridge pattern as reusable hive architecture:** not recommended for us. It is an elegant answer to "no script injection in a hosted chat UI," which is the opposite of our situation. Document it as prior art; do not build on it.

**Repo size:** the ~1.3 GB figure in the brief is not borne out by this clone (~231 MB, ~112 MB of which is `.git` history; remainder is checked-in demo SVGs/MP4s, not build artifacts). A depth-1 clone is small. Not a blocker for adoption.

## Competitive/landscape positioning

Against the heavyweight incumbents, ontosphere sits in a distinct, lighter niche:
- **Protege** (desktop, Java, full OWL editing + reasoner plugins like HermiT/Pellet) and **TopBraid** (commercial, server-backed, SHACL-native) are the authoring/governance heavyweights. ontosphere is far lighter, browser-only, zero-install, and not aiming for their depth of OWL editing or enterprise data management.
- **WebVOWL** is browser-based but is a read-only OWL *visualizer*; ontosphere is a read/write *editor* with reasoning, SHACL, SPARQL, and export, which is a meaningfully larger scope than WebVOWL.
- The genuinely differentiated angle is **client-side OWL 2 DL reasoning via WASM + a designed MCP/agent control surface**. Protege/TopBraid/WebVOWL have no native LLM-agent tool surface; ontosphere was built around one. The combination of (a) fully in-browser, no-backend operation, (b) WASM Konclude DL reasoning, and (c) a 35-tool MCP surface with an LLM-readable operating manual and a vendor-agnostic relay is, as of this recon, a novel positioning. The relay bookmarklet specifically (driving any hosted chat LLM into a live KG editor with no install) is the most original single idea here, even if it is not the right primitive for our MCP-native hives.
