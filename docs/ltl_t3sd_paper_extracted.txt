Abstract
We propose an assertional reinterpretation of the Functionality Cotyledon in A. Wayne Wymore’s Tricotyle-
don Theory of System Design (T3SD). In classical T3SD, functional intent is verified constructively between a
reference system Zspec and implementation Zimpl via Wymore system homomorphism h : Zimpl → Zspec. We
offer an assertional reading of the FC as F CL(Φ) = {Z | Z |=L Φ}, where Φ is a set of temporal logic properties
over Wymore discrete-system trajectories and L is a chosen temporal logic dialect (FO-L TL in the encoding
theorem and case study). We show that any Wymore DSYSTEM admits a canonical FO-L TL encoding ψZ of its
execution semantics, enabling solver-based property checking of Zimpl |=L Φ as an alternative to interactive
homomorphism proofs (in decidable or bounded fragments). The contribution is a reinterpretation of the el-
ements of the Functionality Cotyledon, trace semantics, encoding methodology, and workflow comparison.
We illustrate the comparison on a decentralized UA V swarm with infinite product state and dynamic system
configuration.
Keywords Tricotyledon Theory of System Design (T3SD), First-Order Linear Temporal Logic (FO-L TL), For-
mal Verification, Model-Based Systems Engineering (MBSE).
1 Introduction
Model-Based Systems Engineering (MBSE) promises improved traceability and earlier verification, yet many
requirements remain in natural language or diagrammatic forms without checkable temporal semantics [1].
Checking whether an implementation satisfies stated requirements then depends on manual reconstruction
of intent: systems engineers cannot mechanically determine conformance without informal argument. En-
coding requirements as temporal properties over explicit system trajectories is one path toward model-based
requirements that are formally analyzable and executable rather than purely documentation [1].

Wymore’s Tricotyledon Theory of System Design (T3SD) offers a rigorous foundation for this gap: cotyledon-
structured design spaces, trajectory theorems, and constructive verification via reference systems and homo-
morphisms [2, 3]. Generic model checking lacks T3SD’s design-space semantics; we reinterpret the Func-
tionality Cotyledon (FC) assertionally while preserving cotyledon structure. Constructing and maintaining
homomorphisms is largely manual and poorly matched to evolving implementations, especially when concur-
rent subsystems induce large proof obligations [2]. In our experiments, even modest system homomorphism
arguments span thousands of lines of interactive proof script across conjunctive composition and dynamic
reconfiguration; Section 3.4 contrasts this cost with automated solver-based checking of the same boundary
properties.
We propose encoding functional requirements as a property set Φ in temporal logic dialect L and verifying a
candidate implementation by checking Zimpl |=L Φ over admissible Wymore trajectories. Furthermore, any
Wymore DSYSTEM Z can be translated into an FO-L TL formula ψZ that encodes its execution semantics via
compileSystemFO (Section 3.3), so requirement verification can reuse the same tick-indexed vocabulary as Φ
and, in suitable fragments, automated model checkers or SMT solvers [4, 5]. Assertional FC does not require
a concrete Zspec for checking.
Contributions
• A parameterized assertional reinterpretation of the Functionality Cotyledon, reframing functional veri-
fication as temporal property checking over Wymore trajectories rather than as construction of Wymore
system homomorphisms.
• Trace-based verification semantics for state and output trajectories, aligned with Wymore’s trajectory
theorems [2], supporting model-based requirements executable against open-system boundaries [1].
• A canonical FO-L TL encoding of Wymore DSYSTEM execution semantics and a discussion of solver-based
verification tradeoffs versus interactive homomorphism proofs.
• A workflow comparison of constructive and assertional verification methods on a dynamically configured
UA V swarm system.
The paper proceeds as follows: Section 2 summarizes T3SD, notation, and temporal logics; Section 3 defines
trace semantics, the DSYSTEM-to-FO-L TL encoding, and verification workflows; Section 4 instantiates a UA V
swarm system and compares constructive and assertional workflows; Section 5 relates assertional FC to clas-
sical T3SD. Appendix A gives a full proof of the encoding theorem that takes a DSYSTEM to FO-L TL.
2 Background
2.1 Tricotyledon Theory of System Design
Wymore’s T3SD organizes system design into three cotyledons (design spaces): Functionality (FC), Buildabil-
ity (BC), and Implementability (IC) [2]. The FC contains system designs that satisfy required input/output be-
havior (and, in classical T3SD, preferred performance orderings). BC captures technology constraints and cost
orderings; IC is the space of designs that homomorphically map buildable artifacts to functional ones. In clas-
sical verification, functional intent is realized constructively: system requirements are formalized into a ref-
erence system Zspec (a specification-system 5-tuple in the FC), and implementations are checked via Wymore
system homomorphisms. This paper focuses specifically on the FC.
Figure 1 situates both verification readings within the cotyledon layout: Zspec and property set Φ express func-
tional intent in the FC. Under classical T3SD, a verified implementation lies in the implementability overlap
(i.e., buildable in BC and homomorphic to Zspec), with Zimpl shown in the BC–FC region. Assertional checking
instead asks whether Zimpl |=L Φ and does not require a homomorphism or in fact IC membership. A buildable
system may therefore satisfy Φ while lying outside IC (open question; Section 5.3).
We restrict attention to the subset of system designs that can be expressed within Wymore’s discrete systems
(DSYSTEMS): a deterministic 5-tuple Z = (S, I, O, δ, ρ) with state set S, input set I, output set O, transition
function δ, and readout ρ [2]. Time index t ∈ T advances in discrete, atomic ticks while state components

2.2 Temporal Logics
Temporal logics are formal languages for stating requirements over time-indexed execution traces [6, 4]. They
are used to specify safety (forbidden conditions never occur), liveness (desired conditions eventually occur),
and ordering constraints on behavior.

Table 2 summarizes the operators used below. First-Order L TL (FO-L TL) adds quantifiers over state, input,
or output variables at each tick which are needed when requirements relate continuous variables or indexed
elements. Other dialects (e.g., Signal Temporal Logic (STL) for continuous signals) remain admissible as L.
Because functional intent varies in complexity from discrete event sequencing to continuous signal bounds,
our framework parameterizes the chosen logic as L which allows arbitrarily complex property specification.
2.3 Related Formalisms
Generic state model checking does not, by itself, supply Wymore trajectory semantics or a constructive-versus-
assertional account of T3SD verification. Hybrid and event-scheduled formalisms address dynamics outside
Wymore’s discrete systems with uniform, atomic clock [7, 8].

3.2 Formal Trace Definitions and Observability
For Z = (S, I, O, δ, ρ) and trajectories (f, g, y) as in Section 2, we write g(t) for state and y(t) for output at tick
t ∈ T . Wymore’s standard trajectory construction is:
g(0) = s0 (1)
g(t + 1) = δ(g(t), f(t)) (2)
y(t) = ρ(g(t)) (3)
We define the execution trace induced by initial state s0 and input trajectory f as σ = (f, g, y), where g and y
are given by (1)–(3). Temporal requirements may use mixed observability:
• External (black-box): Φext over outputs y(t) and inputs f(t) only.
• Internal (white-box): properties over state components in g(t).
We also write (g(t), y(t))t∈T when the input component of σ is clear from context.
Under the assertional workflow, systems engineers (1) elicit boundary requirements, (2) encode Φ in dialect L
(optionally splitting Φext and internal properties), (3) obtain Zimpl (possibly via some known system coupling
recipe), and (4) check Zimpl |=L Φ under a stated admissible input class. Admissible input trajectories f are
those consistent with mission protocol and environment assumptions at the system boundary, not arbitrary
elements of I T unless explicitly required [1]. Port-level liveness and coordination belong in Φext; physical
invariants not exported at ports are checked over g. When both Zspec and Φ are present, Φ may abstract the
same mission intent at ports while Zspec supports constructive h; assertional checking does not require Zspec.
Properties over multiple agents are evaluated on the resultant system trajectory, not per-agent in isolation.
We say a 5-tuple Z satisfies Φ, written Z |=L Φ, if for every admissible input trajectory f and initial state s0 in
the stated class, the induced trace σ satisfies σ |=L Φ under dialect L [6, 4]. We write F CL(Φ) = {Z | Z |=L Φ}
for the assertional Functionality Cotyledon. Verifying implementation Zimpl is the membership test Zimpl |=L
Φ. When L is clear we write simply |= Φ. Verification is scoped to Wymore DSYSTEMS at discrete ticks; S, I,
and O may be infinite, and checking strategy is tool-dependent [2, 4]. Subsystems may advance via internal δ
when no external command arrives (i.e., autonomous state transition). We treat such steps as ticks with hold-
last-input or environment-defined f(t), so δ(g(t), f(t)) remains defined. Assertional and classical FC need not
coincide: existence of a homomorphism h guarantees that every implementation trajectory is simulated by a
specification trajectory (i.e., h is a trajectory simulation); Zimpl |=L Φ judges only the variables referenced in
Φ. Two implementations may therefore satisfy the same black-box properties while differing in unobserved

Zspec transition structure (over-specification relative to h).
3.3 Encoding Wymore DSYSTEMs in FO-LTL
Equations (1)–(3) define valid execution of a 5-tuple Z = ( S, I, O, δ, ρ) under input trajectory f and initial
state s0. We separate two FO-L TL roles: an execution encoding ψZ that states what it means for trajectories
(f, g, y) to obey Z, and a requirement set Φ that states mission intent (e.g., φsafe from Section 4). Requirement
checking asks whether implementation trajectories satisfy Φ; the encoding theorem below guarantees that ψZ
captures Wymore dynamics exactly.
Define three formula constructors over trajectory variables g : T → S, f : T → I, and y : T → O:
init(s0) ⇔ g(0) = s0 (4)
step(Z) ⇔ ∀t ∈ T, g (t+1) = δ(g(t), f(t)) (5)
readout(Z) ⇔ ∀t ∈ T, y (t) = ρ(g(t)) (6)
The map compileSystemFO is a structural compiler (not a solver):
compileSystemFO(Z, s0) := init(s0) ∧ step(Z) ∧ readout(Z).
We write ψZ(s0) for compileSystemFO(Z, s0) when s0 is fixed. This use of compile is distinct from encoding
natural-language requirements asΦ or formalizing them asZspec. Autonomous ticks are handled atδ(g(t), f(t))
when f(t) is hold-last-input or environment-defined, consistent with Section 3.2.
A trajectory (f, g, y) is a valid Wymore execution of Z from s0 when it satisfies ψZ(s0). The bi-implication
between valid execution and FO-L TL satisfaction is proved in Appendix A. For strictly finite S, I, and O, a
propositional L TL corollary also exists, but we do not develop it further here.
3.4 Automated Requirement Checking
Step (4) of the assertional workflow (Section 3.2) becomes concrete once Zimpl and/or Φ are fixed. Systems
engineers supply (i) the implementation system 5-tuple, (ii) requirement formulas in L, and (iii) a class of
admissible input trajectories. Checking Zimpl |=L Φ typically proceeds by unrolling δimpl and ρimpl over a finite
horizon (bounded model checking) or by abstracting to a finite-state model, then invoking a model checker or
SMT solver [4, 5]. The compiled execution constraints ψZimpl can be conjoined with Φ in the same tick-indexed
vocabulary: solvers receive ground constraints at each t.
Benefits. Assertional checking does not require constructing or maintaining Zspec or homomorphism h.
When structure changes, systems engineers re-run property checking on the same Φ rather than rebuilding an
interactive homomorphism proof. For safety properties over tick-local state, SMT solvers handle arithmetic
and finite unrollings effectively.
Tradeoffs. Satisfaction of Φ guarantees only what Φ states, not trajectory simulation under h. Universal
quantification over all admissible environment trajectories f and unbounded-time liveness ( F ) are not fully
automatic in general: FO-L TL over infinite state and real-valued coordinates is undecidable [4], and practi-
tioners rely on bounded horizons, abstractions, environment models, or finite-state projections. Propositional
L TL on finite abstractions, on the other hand, is mature and largely automated via model checking [4].
Contrast with proof assistants. Interactive theorem provers (e.g., Lean 4 [9]) yield reusable proof objects:
a verified homomorphism is strong evidence for all trajectories. The engineering cost is upfront proof devel-
opment that scales poorly with conjunctive composition, product state, and dynamic reconfiguration. Solvers

decidable fragments. The encoding ψZ bridges the paradigms by placing Wymore dynamics and requirements

3.5 Constructive Verification Criteria
Classical FC seeks a system homomorphism h : Zimpl → Zspec: surjective maps on states, inputs, and outputs
that preserve δ and ρ [2]. Intuitively, every trajectory of Zimpl is simulated by a corresponding trajectory in
Zspec. When Zimpl is built from concurrent subsystems, Wymore’s conjunctive composition (CSY) forms a
resultant system with shared input ports and per-component readouts [2]; the homomorphism obligation
then relates the composed implementation to a (typically simpler) reference system.

4 Case Study: UAV Swarm
We apply the proposed methodology to an example system in order to compare constructive and assertional
workflows on the same Zimpl. This is done as a conceptual demonstration, not an empirical benchmark.
4.1 Specification System Definition
Under classical T3SD, swarm functional intent is first expressed as a behavioral reference systemZspec (Table 4,
left column; Figure 4). States record aggregate search progress (K, t, a): covered cells K ⊆ Cells over region
R, mission clock t, and phase a. Boundary ports are broadcast MissionCmd and aggregate MissionOut; ρ maps
(K, t, a) to coveragePct, missionActive, deadlineMet, and ack. On tick, t advances and K grows when active
agents enter new cells; liveness requires K = Cells before t > T max while a = searching.
4.2 Implementation System Definition
A decentralized swarm of N UA Vs executes cooperative search over a bounded region R ⊂ R2 while main-
taining minimum separation dsafe. A Return to Base (RTB) event removes a vehicle from the current active set
M ⊆ { 1, . . . , N} when its battery falls below threshold. Participation change is the deliberate confounder:
for constructive proofs, dynamic system configuration may require revisiting a homomorphism h between the
specification and current system configuration, whereas assertional checking re-evaluates the same Φ on the
suffix trace.
The implementation Zimpl is a conjunctive composition (CSY) of N agent subsystems sharing MissionCmd
at the mission bus (Table 4, right column; Figure 3). Assertional specification Φ = {φsafe, φlive} in FO-L TL
provides the alternative functional artifact; port and property detail appear in Section 4.3.

Automation contrast. On the constructive path, a homomorphism proof library for CSY swarm systems is
large (thousands of lines of interactive script in our formalization) and likely must be revised when N , the
active set M , or δimpl changes. On the assertional path, engineers re-encode δimpl and ρimpl as tick constraints
via ψZimpl (Section 3.3) and re-invoke checking: φsafe reduces to tick-local arithmetic over agent coordinates (a
natural SMT task), while φlive may require a bounded or unbounded-time strategy depending on the chosen
tool fragment. We report this as a conceptual workflow contrast, not a timed benchmark.
This case study exercises an infinite state space, continuous coordinates at discrete ticks, FO-L TL with mixed
observability, and participation change as proof-maintenance for h, not as impossibility of classical T3SD.
Achieving dynamic system reconfiguration is possible under classical T3SD, but we expect it to be much more
expensive than the assertional methodology.
5 Discussion
Encoding functional requirements as temporal logic moves intent from natural-language statements into a
checkable formal language [1, 4], enabling explicit semantics for requirements and their execution/analysis
against system trajectories. Assertional property checking aligns with open-system verification practice in
systems engineering [1], though we do not claim normative alignment with any particular process.
T3SD pedagogy traditionally introduces constructive FC first: students formalize Zspec and prove h [2]. Asser-
tional FC is offered here as a reinterpretation of the same Functionality Cotyledon for verification workflow,
not a replacement of its structure or of classical T3SD when a reference system already exists. We suggest
that Wymore’s allowance for not-necessarily-finite state and trajectory-based semantics supports evaluation
of temporal logic at ticks as a dual reading of the same design space.
5.1 Relationship to Classical Functionality Cotyledon
Classical T3SD defines functional designs constructively while assertional FC defines them by temporal prop-
erties in dialectL. Existence of a Wymore system homomorphismh implies trajectory simulation: every imple-
mentation trace maps under h to a valid specification trace. Conversely, Zimpl |=L Φ does not imply existence
of h to a given Zspec (Section 3.2). Logical equivalence between these methods is left to future work.
We also note that constructive verification may still be preferable when |S| is small, Zspec already exists, and
changes are internal refinements with fixed observable behavior.
5.2 Expressivity and Tooling
Assertional FC is intended to capture functional intent through F CL(Φ) without requiring a reference system
Zspec. In decidable or bounded fragments, property checking of Zimpl |=L Φ can be partially automated via
model checkers or SMT solvers. At the theoretical level, L may range from propositional L TL to FO-L TL over
trajectory variables at ticks. Two fragments deserve separate treatment for automation.
Execution-encoding fragment (SMT-friendly). The constructors init, step, and readout in Section 3.3 are
tick-indexed equalities over δ and ρ. Conjoined with a finite unrolling horizon, they map directly to ground
constraints in SMT [5] or to finite-state transition relations for explicit model checking [4]. This fragment is
intentionally aligned with Wymore’s trajectory recurrence.
Requirement fragment (harder). The UA V case study’s Φ adds first-order quantifiers over agents, G/F
temporal operators, and real-valued separation (a fragment that is undecidable in general). Practitioners use

5.3 Limitations and Open Questions
• Equivalence: We have not demonstrated that assertional and classical FC are equivalent; Tables 3–
5 compare workflows, not logical strength (see also Section 5.2). We believe that arbitrarily complex
requirements may always be expressed using some temporal logic, but this is left to future work.
• Implementability vs. assertional membership: Classical IC membership requires a buildable system
with homomorphism to a functional reference. Assertional FC membership requires onlyZimpl |=L Φ. We
have not characterized when a system can lie in BC but outside IC yet still satisfyΦ, nor when Zimpl |=L Φ
implies existence of h to a given Zspec (Figure 1).
• Encoding vs. decidability: The DSYSTEM-to-FO-L TL encoding places dynamics and requirements in
one vocabulary, but it does not make verification decidable. FO-L TL over real-valued coordinates re-
mains undecidable in general [4].
• Infinite state: In the case study, the swarm system may be checked with tick-local SMT for separation
at sampled states, an under-approximation for inter-tick motion [4, 5]. Continuous-time between ticks
is left to future work.
• Constructive tractability: Universal proofs over all participation sets and real trajectories remain open;
our formalization suggests substantial interactive effort even for modest systems.
• Autonomous steps: Complete liveness semantics for ticks without external input is an area for future
theoretical work.
• Empirical / MBSE evidence: We provide no timed benchmark comparing homomorphism effort to prop-
erty checking; qualitative evidence includes proof-library scale and workflow contrast. SysML require-
ments trace and standard tooling integration remain future work.
6 Conclusions
In this paper, we offer an assertional reinterpretation of Wymore’s Functionality Cotyledon: functional ver-
ification as temporal property checking over trajectories, with trace semantics aligned to T3SD. We give a
canonical FO-L TL encoding of DSYSTEM execution and discuss when solver-based checking offers a practical
alternative to interactive homomorphism proofs. The UA V swarm case study illustrates mixed observability,
infinite product state, and RTB participation change, where assertional re-checking of Φ avoids rebuilding
Zspec and h. Logical equivalence between paradigms, autonomous-step semantics, and MBSE tool integration
remain open.

A DSYSTEM to FO-LTL Encoding
This appendix presents the encoding summarized in Section 3.3. The full bi-implication is machine-checked
in a companion Lean 4 library [9].
Given Z = ( S, I, O, δ, ρ) and initial state s0 ∈ S, the compiler produces a formula over trajectory variables
f : T → I, g : T → S, and y : T → O.
Algorithm 1: compileSystemFO
1. CompileInit(s0) ← init(s0), i.e., the constraint g(0) = s0.
2. CompileStep(Z) ← step(Z), i.e., ∀t ∈ T, g (t+1) = δ(g(t), f(t)).
3. CompileReadout(Z) ← readout(Z), i.e., ∀t ∈ T, y (t) = ρ(g(t)).
4. CompileSystemFO(Z, s0) ← CompileInit(s0) ∧ CompileStep(Z) ∧ CompileReadout(Z).
Write (f, g, y) |=FO φ when formula φ holds under the interpretation of Section 3.3. For the atomic construc-
tors:

Conjunction is interpreted componentwise. This matches Wymore’s trajectory equations (1)–(3).
Theorem (execution equivalence). Fix Z, s0, and f . Trajectories (g, y) form a valid Wymore exe-
cution (initial states0, recurrence underδ, readout under ρ) if and only if(f, g, y) |=FO compileSystemFO(Z, s0).
Proof ( ⇒). Let gcanon and ycanon be Wymore’s uniquely defined trajectories from (Z, s0, f). By Wymore’s
trajectory construction, gcanon(0) = s0, gcanon(t+1) = δ(gcanon(t), f(t)), and ycanon(t) = ρ(gcanon(t)). Hence
(f, gcanon, ycanon) satisfies all three conjuncts.
Proof ( ⇐). Suppose (f, g, y) |=FO compileSystemFO(Z, s0). Then g(0) = s0 and g satisfies the step recur-
rence. Wymore’s trajectory uniqueness theorem implies g(t) = gcanon(t) for all t. The readout conjunct gives
y(t) = ρ(g(t)) = ρ(gcanon(t)) = ycanon(t).

A.1 Finite-State Corollary
When |S|, |I|, and |O| are strictly finite, each state, input, and output value can be represented as a propo-
sitional atom at each tick, and step (Z) becomes a finite set of G-guarded implications state s ∧ input i →
X state δ(s,i). A compiled propositional-L TL formula exists for finite Moore machines. However, this is a syn-
tactic expressibility result, not a practical encoder for large |S| · |I|.
