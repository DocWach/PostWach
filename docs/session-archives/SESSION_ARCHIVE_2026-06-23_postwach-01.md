## 4. CRITICAL INCIDENT — my workflow destroyed the principal's manual edits (fixed)

Across many revision rounds I kept a `_build_v7.py` that **regenerated the docx from `V4.docx`** each
iteration and ran **`Get-Process WINWORD | Stop-Process -Force`** before each build to release the file lock.
The principal was editing the live file in Word. Force-killing Word **discards unsaved in-memory edits**, and
the regenerate then overwrote the disk file. The principal lost edits repeatedly and flagged it as "a
consistent problem when editing papers" — and it is already in memory ([[feedback_no_overwrite_manual_edits]],
caught 2026-06-13 on Vision 2035), yet it recurred here, worse.

- **Detection:** a `~$OSW...V3.5.docx` lock file in the folder = Word has the file OPEN right now; the
  on-disk copy lagged the principal's unsaved session. The principal was asked to Ctrl+S before anything
  touched it; edits were saved and the file sent to RTSync.
- **Principal's actual edits (now canonical):** removed the 6pt paragraph space-after throughout (tighter),
  added 12pt space-before on "In sum…" and "The work plan is de-risked…", removed the title's trailing comma.
  No text changes.
- **Fix:** never `Stop-Process WINWORD`/force-kill the app holding a deliverable; once a deliverable is
  hand-edited, abandon the build script and edit surgically; to compare, build a reference copy to a
  DIFFERENT filename and diff, never the live name. Updated `feedback_no_overwrite_manual_edits.md` + the
  MEMORY.md index with this root cause. `_build_v7.py` is **retired** for this deliverable.
